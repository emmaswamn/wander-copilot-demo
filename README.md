# 漫行副驾 · Wander Copilot

> 国内自驾 / 火车 / 飞机旅行规划 AI 助手。表单式信息收集 → 联网景点检索 → 分块生成完整 Markdown 攻略。

---

## 设计思路

### 表单优先

产品没有自由对话输入，而是通过**分步表单**收集规划所需的结构化信息：

| 步骤 | 收集内容 |
|---|---|
| 基本信息 | 出发地、出行人数、出发日期、行程天数、往返时间上限 |
| 交通方式 | 自驾 / 火车 / 飞机（支持多选；自驾额外填写车型与动力类型） |
| 集合地 / 途经 | 可选：人际汇合地、自驾必经途经点 |
| 目的地 | AI 推荐卡片，含交通与住宿参考量级，用户选定一个 |
| 景点选点 | 联网检索该目的地景点列表，用户勾选进攻略 |
| 网络参考 | 可选：联网搜索目的地攻略，勾选摘录作为生成素材 |

所有输入最终序列化为结构化 JSON，注入攻略生成的各个 prompt 块，模型不做自由发挥。

---

## 三种交通方式与攻略生成分流

```
交通方式
  ├── 自驾   → 完整九章（含路线骨架 + 路网摘要）  ← 当前最完善
  ├── 火车   → 七章（无自驾专属章节）
  └── 飞机   → 七章（无自驾专属章节）
```

目前自驾路径经过完整验证；火车 / 飞机路径共享大部分生成逻辑，章节内容会依交通方式调整叙述口径。

## 攻略生成：各章节说明

### 章节总览

| # | 章节名 | 类型 | 自驾专属 |
|---|---|---|---|
| 一 | 行程摘要 | derive（代码直出） | |
| 二 | 行前与费用 | LLM 生成 | |
| 三 | 出行交通 | LLM 生成 | |
| 四 | 玩什么 | LLM 生成 | |
| 五 | 景点票务与配图 | derive（联网检索结果格式化） | |
| **六** | **自驾路线骨架** | **LLM 生成** | **✓** |
| 七 | 按日安排 | LLM 生成 | |
| **八** | **路网与补给摘要** | **LLM 生成（与六合并一次请求）** | **✓** |
| 九 | 免责声明 | derive（代码直出） | |

> 五（景点票务）仅在开启「完整联网」且勾选了景点时出现。

### 自驾专属逻辑

- **车型与动力**（如「小米SU7（纯电）」）写入 trip JSON，进入路线块 prompt，影响充电站建议、补给策略的叙述口径
- **六和八合并一次 LLM 请求**：模型输出两段正文用分隔符隔开，代码侧拆分后分别插入对应位置，减少一次 API 调用
- **拼接顺序**：六 → 七 → 八（生成顺序不同于拼接顺序，v2.2 修复了早期版本的章节乱序问题）

## API 调用与 Token 用量

> 以「2 人自驾 3 天、3 个景点、开启完整联网」为参考样本。

### 调用流程

```
① 并行搜索（火山方舟 Responses API + web_search）
     attraction_fees   × 8 来源   →  票务 JSON
     attraction_images × 5 来源   →  配图 URL JSON

② derive（纯代码，0 次 API 调用）
     title / summary / poi_pricing / disclaimer

③ 分块生成（火山方舟 Responses API，共 5 次）
     trip_cost → transport → play_what → drive_route_road → daily

④ 拼接
     stitchGuideMarkdown：固定 ## 标题 + 模型正文按序合并
```

### Token 分布（自驾典型篇）

| 阶段 | 接口类型 | 约消耗 |
|---|---|---|
| POI 票务联网检索 | Responses + web_search | ~14k |
| POI 配图联网检索 | Responses + web_search | ~10k |
| 分块 LLM 生成（5 次） | Responses API | ~20k |
| 目的地推荐 + 景点拉取 | Chat / Responses | ~8k |
| **自驾单篇合计** | | **~50k～65k tokens** |

非自驾（火车 / 飞机）少一次 `drive_route_road` 生成，约 **~35k～50k tokens**。

## 设计目标

生成与 ChatGPT 同款质量的国内旅游攻略，同时把运行成本压到接近零。

**为什么成本可以接近零：**

火山方舟提供「安心试用」免费资源包；免费资源包生成的内容会进入模型训练数据，但攻略本身不包含任何用户隐私信息（仅含目的地名称、行程天数、交通方式等公开旅行要素），对普通用户完全可接受。在免费额度范围内，单篇攻略的实际 API 成本为 **¥0**。

当前版本为了让任何人都能直接试用，采用 **BYOK（Bring Your Own Key）** 策略——用户自己填写火山方舟 API Key，密钥仅存在本地浏览器，不经过任何服务器。

---

## 部署架构

GitHub Pages 是纯静态托管，浏览器直接调用火山方舟 API 会被 CORS 拦截。当前方案通过 **Cloudflare Workers** 做边缘反代解决此问题：

```
用户浏览器
    ↓  访问
GitHub Pages（静态前端）
    ↓  API 请求（带用户自己的 Authorization Key）
Cloudflare Worker · wander-proxy（CORS 转发，免费 10 万次/天）
    ↓  透传
火山方舟 API（ark.cn-beijing.volces.com）
```

- Worker 只转发 `/api/v3/*` 路径，不存储、不记录任何请求内容
- Key 由用户在请求头里携带，全程不落地任何服务器
- 本地 `npm run dev` 走 Vite proxy，不经过 Worker，行为与生产一致

Worker 源码见 `proxy/`，使用 `wrangler deploy` 一键部署到 Cloudflare 免费层。

## 可优化方向

### Redis 后端缓存（景点与票务）

当前每次攻略生成都会重新调用 LLM 拉取景点列表、票务信息和配图 URL。这部分内容**与目的地强绑定、更新频率低**（景点名称和票价量级通常按季度变化），是最适合缓存的数据。

| 可缓存的调用 | 单次约消耗 | 缓存后节省 |
|---|---|---|
| `city_attractions`（景点 JSON 列表） | ~5k tokens | ✓ |
| `attraction_fees`（票务检索） | ~14k tokens | ✓ |
| `attraction_images`（配图检索） | ~10k tokens | ✓ |
| **三项合计** | **~29k tokens** | |

对热门目的地（大理、张家界、西藏等），缓存命中时每篇可节省约 **25k～30k tokens**，总用量从典型的 60k 压缩至 **30k～35k**，减少约 **50%**。

攻略生成的核心部分（分块 LLM 生成，~20k tokens）因为依赖用户的个性化输入（行程天数、车型、勾选景点、出发时间），无法缓存，是下限。

## 已知限制

- 充电 / 加油在各章节描述口径可能不完全一致（与真人攻略同级，接受）
- 各章节间存在少量内容重叠（块边界所致）
- 所有票价、开放时间均为检索摘要与模型常识，不构成实时承诺，**以官方为准**
